## Findings

### 1. RevenueCat error overlay blocks automation in Expo Go
**Severity:** Low

**What I found:** The RevenueCat "Error configuring Purchases: Invalid API key" console error overlay pops up on cold launch in Expo Go, blocking Maestro automation on subsequent navigation attempts.

**Why it matters:** Test automation requires dismissing this overlay. Real users on a production build will never see it (RC native module works in dev builds). For QA purposes it's noise.

**Expected behavior:** Error caught silently; toast shown at bottom (which is the production behavior).

**Actual behavior:** Expo Go's LogBox displays a full-screen red overlay that intercepts taps.

**Recommended fix:** Not a code fix — this is an Expo Go limitation. For Maestro flows, add `- tapOn: { text: "Dismiss", optional: true }` after `launchApp` to auto-dismiss.

---

### 2. Brand name truncation on Discover Curated Edits cards
**Severity:** Medium (tracked from v2 QA report)

**What I found:** "REGIME DES FLEU..." truncated on compact cards. The letter-spacing fix for brands >15 chars works for "OLFACTIVE STUDIO" but not for "REGIME DES FLEURS" (17 chars uppercase with wide letter-spacing).

**Why it matters:** User sees an incomplete brand name. Minor but noticeable.

**Recommended fix:** Reduce font size by 1pt for brands > 16 chars, or allow `numberOfLines={2}` on brand text.

---

### 3. pgTAP RLS tests not yet run against live Supabase
**Severity:** Medium

**What I found:** `supabase/tests/rls.test.sql` exists with 42 test cases covering all 9 user-facing tables, but has not been executed against a running Supabase instance via `supabase test db`.

**Why it matters:** Tests are written but unverified. A syntax error or migration ordering issue could cause false confidence.

**Recommended fix:** Run `supabase test db` against the local Supabase instance. Fix any failures. This is the one remaining verification for criterion (c).

---

### 4. `checkAndAwardBadges` — newly wired, untested end-to-end
**Severity:** Low

**What I found:** `checkAndAwardBadges()` was imported but never called in `useWearLogStore.add()`. It was wired in this session (called after successful `syncWrite`). However, it hasn't been tested end-to-end with a real user logging multiple wears to hit a streak threshold.

**Why it matters:** Badge awards are a retention mechanic. If `checkAndAwardBadges` fails silently, users won't see streak badges.

**Recommended fix:** Manually test: log 7 wears for the same user, verify `user_badges` row appears for `streak_7`.

---

### 5. Empty pill confirmed on wardrobe screen
**Severity:** N/A (Positive finding)

**What I found:** The new "Empty" filter pill renders correctly on the wardrobe screen alongside All/Want/Have/Worn. Empty state shows "0 empty" in the subtitle when selected. Screenshot confirms: `m2c-wardrobe-empty-pill.png`.
